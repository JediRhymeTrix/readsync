## ⚠ REQUIRED ACTION for the agent that runs `go mod tidy` / final integration

**This must be done once, before Phase 4 (or whichever phase first does a
full `go build ./...` or `go test ./...` on the main module).**

`go.sum` does not exist because `go mod tidy` has never been run with the
new Phase 3 dependencies in `go.mod`. The three new packages need their
checksums recorded:

```
github.com/gin-gonic/gin v1.10.0
golang.org/x/crypto v0.23.0
golang.org/x/time v0.5.0
```

### What to do

On the machine where you first build/test after Phase 3 is merged:

```powershell
# Windows (requires TDM-GCC for go-sqlite3 CGO)
$env:CGO_ENABLED = "1"
go mod tidy
```

```bash
# Linux/macOS CI
sudo apt-get install -y gcc   # or equivalent
CGO_ENABLED=1 go mod tidy
```

`go mod tidy` will:
1. Download `gin` and its transitive dependencies (`bytedance/sonic`,
   `gin-contrib/*`, `go-playground/validator`, etc.)
2. Download `golang.org/x/crypto` and `golang.org/x/time`
3. Write `go.sum` with all checksums
4. Update `go.mod` to add any indirect dependencies gin requires

Then verify Phase 3 builds and tests pass:

```bash
CGO_ENABLED=1 go build ./...
CGO_ENABLED=1 go test -v -count=1 -timeout 60s \
    ./internal/adapters/koreader/...
```

And the conformance replay against the live adapter:

```bash
# Terminal 1 — start dev runner
CGO_ENABLED=1 go run . --db /tmp/test.db

# Terminal 2 — replay KOSync fixtures
bash fixtures/koreader/curl-replay.sh http://localhost:7200
```

Expected output: register 201, auth 200, push 200, pull 200 with
`percentage: 0.47` and a valid `timestamp`.

### Why it is deferred

`go mod tidy` requires network access and a GCC toolchain (for
`go-sqlite3`). AI coding agents in this workflow do not have outbound
network access; the step must be performed by a human developer or a
CI runner with internet access.

No code changes are needed — only running `go mod tidy` and committing
the resulting `go.sum` and updated `go.mod`.
